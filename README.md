## О работе

Цель работы — реализовать классический алгоритм LZ77 (Лемпель–Зив, 1977) и
сопроводить его наглядной интерактивной демонстрацией. Приложение позволяет:

- сжать произвольный текст и **посмотреть, как алгоритм работает шаг за шагом**:
  состояние скользящего окна, найденное совпадение, очередной токен;
- управлять параметрами алгоритма (размеры search- и lookahead-буферов) и
  наблюдать, как они влияют на качество сжатия;
- загрузить файл, получить сжатую версию `.lz77` и разжать её обратно — для
  демонстрации того, что сжатие выполняется без потерь.

## Алгоритм LZ77

LZ77 — словарный алгоритм сжатия без потерь. Основная идея: **если фрагмент
данных уже встречался ранее, его можно заменить ссылкой назад**, вместо того
чтобы повторять символы.

### Скользящее окно

Алгоритм поддерживает «скользящее окно», разделённое на две части:

```
... уже сжатые символы ... ▸ ... ещё не сжатые ...
    ←   Search buffer  →     ←  Lookahead  →
```

- **Search buffer** (N символов) — уже обработанная часть. В ней алгоритм ищет
  повторы. Чем больше N, тем дальше можно искать совпадение, но тем больше бит
  уходит на хранение `offset` в токене.
- **Lookahead buffer** (M символов) — символы, которые ещё предстоит закодировать.
  В нём ищется самый длинный префикс, совпадающий с какой-либо подстрокой
  search-буфера. M ограничивает максимальную длину одного совпадения.

### Шаг алгоритма

На каждом шаге:

1. Среди подстрок search-буфера ищется такая, чей префикс наиболее длинно
   совпадает с началом lookahead-буфера. Результат — пара `(offset, length)`.
2. Берётся следующий символ за совпадением — `next_char`. Он нужен, чтобы
   алгоритм мог двигаться вперёд даже когда совпадения нет
   (в этом случае токен выглядит как `(0, 0, c)`).
3. На выход идёт **токен** `(offset, length, next_char)`.
4. Курсор сдвигается на `length + 1` позицию.

Совпадение допускается перекрывающее: оно может «заезжать» в lookahead-буфер.
Так компактно кодируются длинные повторы вроде `aaaaaaaa`.

### Декодирование

Декодер тривиален: проходит по списку токенов и для каждого копирует `length`
символов с позиции `(текущая_длина − offset)`, затем добавляет `next_char`.
Никаких словарей или дополнительных таблиц хранить не требуется.

## Реализация

### Файл `lz77.py`

Самостоятельный модуль без внешних зависимостей. Ключевые функции:

- `encode(data: str, search_size, lookahead_size)` — сжимает строку, возвращает
  словарь с токенами, **пошаговой трассировкой** (состояние буферов на каждом
  шаге) и статистикой (размер, коэффициент сжатия, число бит на токен).
- `decode(tokens)` — восстанавливает исходную строку из списка токенов.
- `encode_bytes(data: bytes, …)` / `decode_bytes(tokens)` — обёртки для работы
  с произвольными байтами. Используется приём с кодировкой **latin-1**, которая
  является биекцией между `bytes` и `str` (1 байт ↔ 1 символ Unicode 0–255), что
  позволяет переиспользовать строковый алгоритм для любых файлов.
- `pack(result)` / `unpack(blob)` — сериализация токенов в бинарный формат
  `.lz77` через `struct`.

Поиск самого длинного совпадения реализован простым перебором всех позиций
search-буфера. Это O(N·M) на каждый шаг — асимптотически не оптимально, но
прозрачно и достаточно для демонстрационных размеров и файлов до нескольких МБ.

### Файл `main.py`

FastAPI-приложение с четырьмя эндпоинтами (см. ниже) и отдачей статики из
каталога `static/`. Ограничение на размер загружаемого файла — 5 МБ (для защиты
от случайной загрузки больших архивов).

### Фронтенд (`static/`)

Сделан на ванильном JS без бандлеров и фреймворков. Тёмная тема в стиле
современных IDE (палитра — Tokyo Night). Возможности:

- две вкладки — «Текст» и «Файл»;
- пошаговая визуализация скользящего окна с цветовой подсветкой:
  search-буфер серым, совпадение жёлтым, курсор голубым, следующий символ красным;
- управление шагами — кнопки и горячие клавиши `←` / `→` / `Space`;
- список токенов с кликабельной навигацией;
- мини-статистика (исходный/сжатый размер, экономия, число токенов);
- график сравнения размеров (Chart.js);
- drag-and-drop загрузка файлов и скачивание `.lz77`;
- модальное окно «Как работает LZ77» с теорией;
- адаптивная вёрстка для разных размеров окна.

## REST API

| Метод | Путь | Описание |
|-------|------|----------|
| `POST` | `/api/encode` | Сжимает текст из JSON, возвращает токены + пошаговую трассировку + статистику |
| `POST` | `/api/decode` | Восстанавливает текст из списка токенов |
| `POST` | `/api/compress-file` | Принимает файл (multipart), возвращает бинарный `.lz77` |
| `POST` | `/api/decompress-file` | Принимает `.lz77`, возвращает восстановленный файл |

Интерактивная документация Swagger доступна по адресу
<http://127.0.0.1:8000/docs> (генерируется FastAPI автоматически).

## Бинарный формат `.lz77`

Формат использует **побитную упаковку**: ширина полей `offset` и `length`
зависит от размеров буферов и берётся минимально достаточной.

```
Заголовок (12 байт, byte-aligned):
+----+----+----+----+----+----+----+----+----+----+----+----+
| 'L'| 'Z'| '7'| '7'|  search_sz  |  lookahead  | token_count|
+----+----+----+----+-------------+-------------+------------+
        magic            u16 BE        u16 BE        u32 BE

Поток токенов (плотно упакованные биты, MSB-first):
| offset | length | hn | next | offset | length | hn | next | ...
   k_o     k_l     1     8      k_o     k_l     1     8

где:
   k_o = ⌈log2(search_sz + 1)⌉   бит на смещение
   k_l = ⌈log2(lookahead + 1)⌉   бит на длину
   hn  = флаг наличия next_char (1 бит)
   next = 8 бит, присутствует только если hn = 1
```

При `search=4096, lookahead=64` каждый токен занимает 13 + 7 + 1 + 8 = **29 бит**
(вместо 48 бит при наивной упаковке в целые байты). На реальных файлах это
даёт прирост коэффициента сжатия в полтора-два раза.

**Замечание о малых файлах.** Заголовок добавляет 12 байт фиксированного
накладного расхода, плюс каждый литерал (токен без совпадения) стоит ~22 бита.
Поэтому файлы меньше нескольких сотен байт могут после «сжатия» вырасти —
это нормальное свойство любого алгоритма с фиксированной структурой токена.
